# Документы онбординга: список + лёгкие шаблоны

Цель: собрать “достаточно” информации для **оценки денег/сроков и рисков** без детальной техдокументации.

Использование:
- заполняйте по ходу онбординга, не обязательно всё сразу
- допускается объединять документы (например, `Estimate` + документ про риски/допущения/исключения)
- если пункт не влияет на оценку/риски/согласование — пропускайте

## Обязательные документы (для типового корпсайта/лендинга)

### 1) Brief / One‑pager

**Зачем**: быстрый общий контекст и выравнивание ожиданий.

Шаблон:
- Клиент / бренд:
- Что продаём/предлагаем (1–2 предложения):
- География / языки:
- Целевая аудитория (кратко):
- Главная цель сайта:
- Топ‑3 конкурентных преимущества:
- Референсы (нравится/не нравится) + почему:
- Ограничения (срок, бюджет‑диапазон, платформа, контент):
- Контакты decision maker:

### 2) Goals & Metrics (легкая версия)

**Зачем**: понимать, что считать успехом и что оптимизировать.

Шаблон:
- Цель #1:
  - Метрика:
  - Как измеряем:
  - Целевое значение/ориентир (если есть):
- Цель #2:
  - Метрика:
  - Как измеряем:

### 3) MoSCoW Scope (+ Not in scope)

**Зачем**: управлять фикс‑прайсом и защититься от расползания.

Шаблон:
- Must:
  - …
- Should:
  - …
- Could:
  - …
- Not in scope:
  - …

Правило изменения (коротко, 2–4 строки):
- Что считаем изменением:
- Как пересогласуем срок/стоимость:

### 4) Sitemap (карта страниц)

**Зачем**: согласовать структуру и объём работ “по страницам”.

Шаблон:
- Главная
- О компании
- Услуги:
  - Услуга 1
  - Услуга 2
- Кейсы
- Блог/Новости (если нужно)
- Контакты
- Политики/условия (если нужно)

### 5) Content requirements (требования к контенту)

**Зачем**: убрать главный риск сроков — отсутствие материалов.

Шаблон:
- Кто готовит контент: (клиент/вы/копирайтер)
- Сроки предоставления:
- Для каждой ключевой страницы:
  - Текст: кто/когда
  - Фото/видео: кто/когда
  - Кейсы/отзывы/сертификаты: кто/когда
- Доступы к материалам (диск/папка/брендбук):

### 6) Estimate (сроки/деньги)

**Зачем**: дать понятную оценку и связать её с допущениями.

Шаблон:
- Итоговая стоимость: …
- Итоговый срок: …
- Разбиение по этапам (очень крупно):
  - Дизайн: …
  - Разработка: …
  - QA/правки: …
  - Релиз: …
- Что влияет на срок:
  - контент
  - скорость ответов/согласований
  - доступы/интеграции

### 7) Risks, Assumptions & Exclusions (риски, допущения и исключения)

**Зачем**: честно фиксировать неизвестности, управлять ожиданиями и зафиксировать рамку фикс‑прайса.

Шаблон:
- Допущения (что считаем правдой для оценки):
  - …
- Риски (что может сорвать срок/увеличить стоимость/качество):
  - Риск:
    - Вероятность (низк/средн/высок):
    - Влияние (низк/средн/высок):
    - Что делаем для снижения:
- Быстрые проверки (что выясняем первым):
  - …
- Исключения / не входит:
  - …
- На стороне клиента (ответственность клиента):
  - контент / реквизиты / доступы / утверждения
- Лимиты (если применимо):
  - количество итераций правок по дизайну/контенту
  - сроки ответов

### 8) Decision log (журнал решений) — желательно

**Зачем**: экономит время на “мы так не договаривались”.

Шаблон (по строке на решение):
- Дата:
- Решение:
- Почему:
- Кто утвердил:

## Опциональные документы (по ситуации)

### A) References (likes/dislikes)

**Когда нужно**: если дизайн — слабое место или много вкусовых споров.

Шаблон:
- Нравится: ссылка → почему
- Не нравится: ссылка → почему
- Тональность: строгий/дружелюбный/премиум/…

### B) Page blocks outline (скелет ключевых страниц)

**Когда нужно**: если важно быстро “сойтись” по структуре без прототипов.

Шаблон для страницы:
- Страница:
- Цель страницы:
- Блоки сверху вниз:
  - Блок: назначение / контент / CTA

### C) Integrations list

**Когда нужно**: если есть CRM/платежи/чат/аналитика/почта/мессенджеры.

Шаблон:
- Интеграция:
- Зачем:
- У кого доступ:
- Критичность (Must/Should):
- Риски/ограничения:

### D) Existing system inventory (если есть legacy)

**Когда нужно**: редизайн/миграция/SEO‑риски.

Шаблон:
- Домен/регистратор:
- Хостинг:
- CMS/конструктор:
- Аналитика:
- Формы/почта:
- Текущее SEO (что важно сохранить):
- Где хранится контент:

### E) Content plan (если контент “бутылочное горлышко”)

**Когда нужно**: если клиент не готовит контент в срок.

Шаблон:
- Что нужно подготовить:
- Кто делает:
- Дедлайн:
- Статус:
