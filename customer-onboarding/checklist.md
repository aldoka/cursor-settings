# BA чек‑лист онбординга (websites)

Используйте как контрольный список, чтобы собрать “достаточно” информации **без тяжёлой документации**.

Принцип: если пункт не влияет на **оценку/риски/согласование**, его можно пропустить.

## 1) Стейкхолдеры и коммуникация

- [ ] Decision maker определён (один человек)
- [ ] Канал связи и время ответов согласованы
- [ ] Кто поставляет контент (клиент/вы/третьи лица) определено
- [ ] Процесс согласования понятен (сколько итераций правок по дизайну/контенту допустимо)

## 2) Проблема → цель → ценность

- [ ] Сформулирована бизнес‑цель сайта (1–2 предложения)
- [ ] Сформулирована целевая аудитория/сегменты
- [ ] Определено ключевое действие пользователя (lead/звонок/заявка/покупка/подписка)
- [ ] Есть 2–5 метрик успеха (и “как будем мерить”)

## 3) Сценарии (легкая версия user journey / JTBD)

- [ ] Топ‑3 сценария пользователя описаны шагами
- [ ] На каждом сценарии понятны входы/выходы (что увидел → что понял → что сделал)
- [ ] Определены “точки доверия” (кейсы, отзывы, гарантии, цифры, сертификаты)

## 4) Требования и нарезка (slicing)

- [ ] Список требований описан в формате “что + зачем”
- [ ] Требования нарезаны на маленькие проверяемые куски (страницы/блоки/интеграции)
- [ ] Есть явные ограничения (сроки, бюджет, платформа, хостинг)

## 5) Приоритизация MoSCoW

- [ ] Must: минимально необходимое ядро согласовано
- [ ] Should/Could: зафиксированы как “после Must”
- [ ] Not in scope: явно записано, что не делаем
- [ ] Определено, что считается Change Request (простое правило)

## 6) Структура сайта и контент

- [ ] Sitemap согласован
- [ ] По ключевым страницам понятны блоки и их цель
- [ ] Контент‑потребности зафиксированы (какие материалы нужны)
- [ ] Сроки предоставления контента согласованы (или контент берёте на себя)

## 7) Интеграции и зависимости (если есть)

- [ ] Список интеграций есть (CRM, почта, мессенджеры, платежи, карты, чат)
- [ ] Понятно, кто даёт доступы и когда
- [ ] Если legacy: инвентаризация текущего сайта/доменов/аналитики сделана

## 8) Нефункциональные требования (NFR)

- [ ] SEO критично/некритично (и что именно важно)
- [ ] Скорость важна (страницы/метрики ожиданий)
- [ ] Мультиязычность нужна/не нужна
- [ ] Админка нужна/не нужна (и кто будет контент менять)

## 9) Риски и допущения

- [ ] Список допущений составлен (то, что считаем правдой для оценки)
- [ ] Топ‑риски выписаны (контент, согласования, доступы, интеграции, юридическое)
- [ ] Есть план закрытия неизвестностей (что выясняем первым)

## 10) Оценка

- [ ] Оценка по срокам и деньгам готова
- [ ] В оценке отражены допущения/исключения
- [ ] Клиент подтвердил понимание “что входит / что не входит”

## 11) Минимум юридического (РБ)

- [ ] Определено: собираем ли персональные данные (формы/заявки)
- [ ] Определено: нужны ли публичные условия/оферта (если есть продажи/заказы)
- [ ] Определено: кто предоставляет и утверждает юридические тексты
