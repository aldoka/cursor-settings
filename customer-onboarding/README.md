# Customer onboarding (websites)

Этот набор документов предназначен для онбординга клиентов на разработку сайтов (чаще всего: **корпсайт** и **лендинг**, реже: **каталог**, ещё реже: **веб‑сервис/ИИ‑интеграции**).

Контекст и ограничения (под ваш кейс):
- чаще **фикс‑прайс** (диапазон цены уже согласован до онбординга)
- обычно **1 decision maker**, общение напрямую
- на этапе онбординга **не делаем детальную техдокументацию**
- **не используем пакеты** (MVP/Standard/Pro) в продажах на данном этапе
- **не делаем** отдельные документы про ПДн / RACI
- из опциональных артефактов **не используем** прототип и acceptance criteria

## Как пользоваться (быстрый поток)

0) Если это созвон — откройте `onboarding-script.md` и ведите разговор по нему (адаптируя под выбранную стратегию).
1) Выберите стратегию из `01-strategies.md` (можно комбинировать, но назначьте одну «ведущую»).
2) Пройдите задачи до старта из `02-before-development.md` и по ходу заполните документы из `03-documents.md`.
3) Используйте чек‑лист BA из `checklist.md`, чтобы ничего не упустить и быстро резать скоуп через MoSCoW.
4) Если клиент из РБ — проверьте минимальные юридические моменты по `04-legal-belarus.md` (для других регионов — TODO).

## Выход онбординга (Definition of Done)

Онбординг считается завершённым, когда у вас есть:
- **оценка по деньгам и срокам** (с допущениями)
- **риски/неизвестности** и план их закрытия (что выясняем первым)
- согласованный **MoSCoW‑скоуп** (что делаем / что не делаем)
- согласованные **структура сайта и контент‑потребности** (кто и когда поставляет)
- базовые **юридические условия** для запуска (минимум для РБ)
